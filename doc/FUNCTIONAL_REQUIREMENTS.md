# 系統功能需求文件

## 1. 文件目的

本文件用來整理新版系統在重構前需要確認的功能需求。目標是把目前既有專案、技術討論與 prototype 驗證結果整理成一份可用於後續設計、拆工與驗收的基準文件。

本系統的核心目標是：將原本需要在 CAD 軟體中手動完成的鷹架設計稿建立流程，轉換成可由 Web 工具輔助產生的流程，並輸出可在 CAD/DXF 編輯器中後續調整的 DXF 檔。

## 2. 系統目標

系統需要支援以下目標：

1. 讓使用者依據鷹架建立原則輸入基本尺寸與格數。
2. 在 Web 介面中建立、編輯與預覽鷹架配置。
3. 支援平面圖與立面圖等多視圖的產生。
4. 支援常見鷹架元件與標記，例如高度、樓梯、錨點、橫桿、斜撐。
5. 輸出可再編輯的 DXF 設計稿。
6. 優先確保 LibreCAD 可開啟並顯示中文字
7. 系統可部署在 Linux 雲端環境。
8. 後續架構需利於維護、測試與擴充。

## 3. 使用者角色

### 3.1 設計/繪圖人員

主要使用者。需要透過系統快速建立鷹架設計草稿，並下載 DXF 後在 CAD 軟體中進一步調整。

### 3.2 管理/審核人員

可能會檢查輸出的圖面是否符合基本配置、標記與工程需求。此角色目前不是第一階段主要操作對象，但系統產出的 DXF 應具備可讀性與可追溯性。

### 3.3 系統維護者

負責維護鷹架規則、DXF 輸出格式、部署環境與後續功能擴充。

## 4. 主要使用流程

### 4.1 建立新設計稿

使用者可以建立新的鷹架設計稿，並輸入基本資訊：

- 案件名稱。
- 單元長度。
- 單元寬度。
- 單元高度。
- X 方向格數。
- Y 方向格數。
- Z 方向層數。
- 基本規則設定，例如橫桿規則或錨點規則。

系統應根據輸入資料建立初始格線與資料模型。

### 4.1.1 鷹架單元方向定義

鷹架的主要目的，是依據施工中建築物外側建立包覆式設計稿。因此每個鷹架單元不只是單純的長寬高方塊，還需要明確定義哪一個面是「面向建築物」或「沿建築物立面展開」的主要面。

以單元尺寸：

```text
長 180
寬 90
高 210
```

為例，視圖定義應注意：

- 正視圖應主要呈現 `長 x 高`，也就是 `180 x 210`。
- 側視圖應呈現側向深度關係，通常為 `寬 x 高` 或依既有程式定義的側向投影。
- 平面圖應呈現 `長 x 寬`，也就是 `180 x 90`。

原始程式中已經有依照長寬比較與五視圖排列進行處理的邏輯。重構時需要保留這個方向概念，不能只用抽象 X/Y/Z 直接推圖，否則正視圖、側視圖與建築物面的關係可能會錯。

後續模型建議明確加入方向語意：

```text
base.long       沿建築物立面方向
base.width      往外伸出/離建築物深度方向
base.height     垂直高度方向
```

若未來需要支援轉角、內縮、外凸或不同立面方向，應在 layout model 中另外加入面向或座標方向設定。

### 4.2 編輯平面配置

使用者可以在 Web 介面中編輯平面配置。

系統應支援：

- 顯示 X/Y 格線。
- 選取單一格子。
- 多選格子。
- 調整局部格子的高度。
- 標記樓梯位置。
- 標記錨點位置。
- 標記橫桿。
- 標記斜撐。
- 清除指定格子的標記。
- 顯示目前選取狀態與目前工具。

### 4.3 產生多視圖

系統應根據目前鷹架資料模型產生多視圖預覽。

至少需要包含：

- 平面圖。
- 正立面圖。
- 左側立面圖。
- 右側立面圖。
- 背立面圖。

各視圖應由同一份鷹架資料模型推導產生，避免前端預覽與後端 DXF 輸出邏輯不一致。

### 4.4 下載 DXF

使用者完成配置後，可以下載 DXF。

系統應：

1. 將目前設計資料整理成 scaffold model。
2. 由 layout engine 產生圖面座標。
3. 由 DXF renderer 產生 DXF。
4. 回傳可下載檔案。

DXF 需使用 CAD 原生實體，而不是僅將 SVG path 轉成 DXF。

### 4.5 暫存、儲存與匯出歷史

設計稿會經過多次編輯，不應只在最後匯出 DXF 時才保存資料。系統需要區分「尚未匯出的暫存稿」與「已匯出的正式檔案」。

第一階段應支援：

- 建立暫存設計稿。
- 儲存目前編輯狀態。
- 管理歷次儲存紀錄。
- 管理歷次匯出紀錄。
- 從歷史紀錄重新開啟設計稿。
- 匯出 DXF 時建立獨立的匯出紀錄。

檔案命名需要區分暫存稿與已匯出檔：

```text
draft_{caseName}_{timestamp}
export_{caseName}_{timestamp}_{sequence}.dxf
```

或使用資料庫 ID 管理：

```text
DesignDraft
  id
  caseName
  status: draft | exported | archived
  currentVersionId

DesignVersion
  id
  designId
  versionNumber
  scaffoldModel
  createdAt

ExportRecord
  id
  designId
  versionId
  fileName
  filePath
  exportedAt
```

在正式匯出 DXF 前，所有編輯狀態應視為暫存資料；匯出的 DXF 應與暫存稿有不同命名與不同紀錄，避免使用者混淆目前正在編輯的版本與已交付/已下載的檔案。

## 5. 編輯器功能需求

### 5.1 工具模式

編輯器應採用工具模式。使用者選擇工具後，滑鼠點擊或鍵盤確認會依目前工具對格子套用行為。

第一階段建議工具：

| 工具 | 說明 |
| --- | --- |
| 選取 | 選取單格或多格 |
| 高度 | 調整選取格子的高度 |
| 樓梯 | 切換樓梯標記 |
| 錨點 | 切換錨點標記 |
| 橫桿 | 切換橫桿標記 |
| 斜撐 | 切換斜撐標記 |

### 5.2 快捷鍵

系統應支援鍵盤快捷鍵，以降低重複滑鼠操作。

第一階段建議快捷鍵：

| 快捷鍵 | 功能 |
| --- | --- |
| `V` | 選取 |
| `H` | 高度 |
| `S` | 樓梯 |
| `A` | 錨點 |
| `R` | 橫桿 |
| `B` | 斜撐 |
| `Shift + 點擊` | 多選 |
| `Enter` / `Space` | 套用目前工具 |
| `Ctrl + Z` | 復原 |
| `Ctrl + Y` | 重做 |
| `Delete` | 清除選取格子的標記 |
| `Esc` | 取消目前操作 |

### 5.3 復原與重做

系統應支援基本 undo/redo。

需支援的操作至少包含：

- 高度變更。
- 樓梯標記變更。
- 錨點標記變更。
- 橫桿標記變更。
- 斜撐標記變更。
- 清除標記。

建議採用 command history 設計，以便後續支援更複雜的批次操作。

### 5.4 狀態顯示

畫面應顯示必要狀態：

- 目前工具。
- 目前選取數量。
- 已標記高度數量。
- 樓梯數量。
- 錨點數量。
- 橫桿數量。
- 斜撐數量。
- 最近操作訊息。

## 6. 資料模型需求

系統需要明確定義前後端共用的鷹架資料模型。

第一階段資料應至少包含：

```text
ScaffoldModel
  fileName
  base
  role
  grid
  fiveViewGrid
  height
  stair
  anchor
  rung
  bracing
  description
```

### 6.1 base

基本尺寸與格數：

- `long`
- `width`
- `height`
- `coordX`
- `coordY`
- `coordZ`
- `scaling`

### 6.2 grid / fiveViewGrid

格線資料應包含：

- 格子 ID。
- 視圖 ID。
- 座標。
- 尺寸。
- 所屬視圖。

### 6.3 標記資料

系統需要能記錄：

- 高度標記。
- 樓梯標記。
- 錨點標記。
- 橫桿標記。
- 斜撐標記。

後續重構時應避免讓資料只存在 DOM 或 SVG attribute 中，所有設計狀態都應回到資料模型。

## 7. DXF 輸出需求

### 7.1 基本要求

DXF 輸出應符合以下要求：

- 可被 LibreCAD 開啟。
- 優先保留 CAD 實體可編輯性。
- 線段、矩形、圓、文字應為 CAD 原生 entity。
- 不應以 SVG path 轉換作為主要輸出方式。
- 應使用明確圖層。
- 應使用明確文字樣式。
- 視圖方向需符合鷹架面向建築物的語意，正視圖應以沿建築物方向的長度與高度為主。

### 7.2 圖層需求

第一階段建議圖層：

| 圖層 | 用途 |
| --- | --- |
| `VIEW_FRAME` | 視圖外框或輔助框 |
| `GRID` | 格線與主要矩形 |
| `TEXT` | 文字 |
| `ANCHOR` | 錨點 |
| `STAIR` | 樓梯 |
| `RUNG` | 橫桿 |
| `BRACING` | 斜撐 |

### 7.3 文字需求

DXF 中的中文應優先保留為可編輯文字。

要求：

- 使用 `TEXT` 或簡單 `MTEXT`。
- 定義固定文字樣式，例如 `CJK_TEXT`。
- 優先測試 LibreCAD 可用的 `.lff` 字型，例如 `unicode.lff`。
- 避免複雜 MTEXT 控制碼。
- 避免依賴精準字寬排版。

若未來需要更高顯示保證，可考慮額外產生 fallback layer：

```text
TEXT_EDITABLE
  可編輯文字

TEXT_OUTLINE_FALLBACK
  文字外框，預設關閉
```

### 7.4 DXF 版本

目前 prototype 使用：

```text
DXF R2007 / AC1021
```

原因是需要優先處理 Unicode 中文字。若後續需要支援更舊版本，例如 R2000/R2004，需另外驗證中文 escaping 與 LibreCAD 開檔相容性。

## 8. LibreCAD 中文相容需求

第一階段相容目標：

- LibreCAD 可以開啟 DXF。
- 中文文字可以顯示。
- 中文文字盡量保留可編輯。
- 圖層可以被辨識。
- 基本線段與標記可以被選取與修改。

已知風險：

- 若 LibreCAD 找不到指定 `.lff` 字型，中文字可能顯示為方塊或缺字。
- 不同 CAD 軟體的文字 fallback 行為不一致。
- 實際 GUI 開檔測試仍需要在可安裝 LibreCAD 的環境補做。

## 9. 後端技術需求

目前建議方向：

```text
Python + FastAPI + ezdxf
```

原因：

- 適合 Linux 雲端部署。
- `ezdxf` 適合建立 CAD 原生 DXF entity。
- 適合先建立獨立 DXF renderer。
- 容易做資料驗證與自動測試。

若未來需求變成 AutoCAD plugin 或深度整合 AutoCAD .NET API，再評估 C#。

## 10. 非功能需求

### 10.1 可維護性

系統應拆分為清楚模組：

- 前端 editor state。
- 前端 renderer。
- scaffold model。
- layout engine。
- DXF renderer。
- API handler。

不應讓大型單一檔案同時負責狀態、畫面、滑鼠事件、資料轉換與 API 呼叫。

### 10.2 可測試性

系統應具備命令列驗證能力。

目前 prototype 已驗證：

- DXF renderer 可產生並讀回 DXF。
- DXF 中包含必要圖層、文字樣式與中文文字。
- Editor state 支援快捷鍵、選取、套用工具、復原與重做。

後續正式重構應保留類似驗證流程。

### 10.3 部署性

系統應可部署於 Linux 雲端環境。

注意事項：

- 不應依賴雲端主機安裝 GUI CAD 軟體。
- DXF 產生應在後端服務中完成。
- 中文顯示不應依賴伺服器端 GUI 字型渲染。
- 若需要產生 PDF/PNG 預覽，需另行處理 Linux 字型與渲染環境。

### 10.4 相容性

第一階段主要相容目標：

1. LibreCAD。
2. AutoCAD 或其他 DXF 編輯器作為次要驗證目標。

## 11. 第一階段範圍

第一階段重構建議包含：

1. 定義正式 scaffold model。
2. 建立 Python DXF renderer。
3. 建立 API endpoint 產生 DXF。
4. 前端導入 editor state 架構。
5. 支援基本工具模式與快捷鍵。
6. 產出基本五視圖。
7. 支援基本圖層與中文字樣式。
8. 建立自動驗證腳本。

## 12. 暫不納入第一階段的項目

以下項目可延後：

- 完整 AutoCAD plugin。
- DXF 文字外框 fallback layer。
- PDF/PNG 正式輸出。
- 使用者帳號與權限管理。
- 案件版本管理。
- 雲端檔案儲存。
- 多人協作。
- 完整圖框與出圖格式管理。
- 複雜尺寸標註與工程計算。

## 14. 重構前待確認事項

在正式開始重構前，建議最後確認：

1. 第一階段是否以 LibreCAD 作為主要 DXF 相容目標。
2. DXF 是否接受使用 R2007 / AC1021 作為初始輸出版本。
3. 中文字型是否先以 `unicode.lff` 作為主要測試樣式。
4. 是否先採用 Python + FastAPI + ezdxf 作為後端方向。
5. 前端是否先保留 SVG 編輯器形式，不導入大型 CAD canvas library。
6. 第一階段是否只處理單機單使用者流程，不處理帳號與雲端儲存。
